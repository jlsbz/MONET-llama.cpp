# Android 手机端 LLaMA 部署教程（基于 llama.cpp）

汇总整个方案思路，形成一个新的文档

添加代码修改说明

添加其他量化模型的比较（q3 q6）

添加benchmark结果

本文档介绍如何将 LLaMA2-7B 类模型部署到 Android 手机端，并使用 `llama.cpp` 进行本地推理和 benchmark 测试。

本流程适用于：

- Windows 主机
- WSL2 Ubuntu
- Android Studio + Android NDK
- Android 手机
- ADB / Termux
- GGUF 模型格式
- Q4_K_M 等低比特量化模型

---

# 1. 整体思路

我们采用如下部署路径：

```text
HuggingFace 模型
        ↓
转换为 GGUF
        ↓
量化为低比特模型
        ↓
使用 Android NDK 编译 llama.cpp
        ↓
将模型和可执行文件推送到手机
        ↓
在 Android 端运行推理与 benchmark
```

# 2. 环境分工

本项目推荐采用如下分工：

```
Windows
  ├── Android Studio
  ├── Android SDK / Platform Tools
  ├── Android NDK
  ├── ADB
  └── Android 交叉编译

WSL Ubuntu
  ├── llama.cpp 源码
  ├── Python 模型转换环境
  ├── GGUF 转换
  └── 模型量化

Android Phone
  ├── llama-cli
  ├── llama-bench
  └── GGUF 量化模型
```

这样做主要是因为设备限制问题

# 3. WSL 中准备 llama.cpp 环境

进入 WSL Ubuntu 后，安装必要工具：
```
sudo apt update

sudo apt install -y \
  git \
  cmake \
  ninja-build \
  build-essential \
  python3 \
  python3-pip \
  python3-venv
```

并下载 llama.cpp：

···
cd ~
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
···

# 4. 创建 Python 虚拟环境
cd ~/llama.cpp

python3 -m venv .venv
source .venv/bin/activate

pip install -U pip
pip install -r requirements.txt

为什么要创建虚拟环境：

Ubuntu 24.04 之后，系统 Python 通常启用了 PEP 668 保护，直接执行 pip install 可能出现：

error: externally-managed-environment

使用虚拟环境可以避免污染系统 Python，也能保证 llama.cpp 的 Python 依赖被安装在当前项目中。

以后每次运行模型转换脚本前，建议先执行：

cd ~/llama.cpp
source .venv/bin/activate
# 5. 将 HuggingFace 模型转换为 GGUF

假设 HuggingFace 格式的 LLaMA2-7B 模型位于：

/mnt/d/models/llama2-7b-hf

执行：

cd ~/llama.cpp
source .venv/bin/activate

python3 convert_hf_to_gguf.py \
  /mnt/d/models/llama2-7b-hf \
  --outtype f16 \
  --outfile llama2-7b-f16.gguf

为什么要转换：

llama.cpp 不直接读取 HuggingFace 的 safetensors 或 PyTorch 权重，而是使用 GGUF 格式。GGUF 会将模型权重、结构信息、tokenizer 信息等打包成一个更适合本地推理的格式。

为什么先输出 F16：

F16 GGUF 是一个较完整的中间格式，后续可以基于它量化成不同格式，例如：

Q4_K_M
Q5_K_M
Q6_K
Q8_0

如果直接生成某种量化格式，后续想换量化格式会不方便。

是否换新模型需要重新执行：

是。只要换了新的 HuggingFace 模型、checkpoint 或结构，都需要重新执行本步骤。

# 6. 编译 WSL 版 llama.cpp 工具
cd ~/llama.cpp

cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build -j4

编译完成后，会生成：

build/bin/llama-cli
build/bin/llama-bench
build/bin/llama-quantize

为什么要先编译 WSL 版：

这里主要是为了得到 llama-quantize，用于在 WSL 中对 GGUF 模型进行量化。这个版本是 Linux x86_64 可执行文件，只用于主机端模型处理，不用于手机端运行。

是否换新模型需要重新编译：

不需要。模型变化不会影响工具本身。只有在更新 llama.cpp 源码，或者修改了源码后，才需要重新编译。

# 7. 量化模型

推荐先使用 Q4_K_M：

cd ~/llama.cpp

./build/bin/llama-quantize \
  llama2-7b-f16.gguf \
  llama2-7b-q4_k_m.gguf \
  Q4_K_M

为什么要量化：

LLaMA2-7B 的 F16 模型约 13GB，手机端通常无法承受这样的内存占用和带宽开销。量化可以显著降低模型大小和访存压力。

Q4_K_M 通常约为 4GB，是手机端部署 7B 模型时比较常用的折中选择。


为什么选择 Q4_K_M：

Q4_K_M 在模型大小、推理速度和输出质量之间比较平衡，适合移动端 demo 和初步 benchmark。

是否换新模型需要重新执行：

是。每个新模型都需要重新量化。如果只是更换量化格式，也需要重新执行本步骤。

# 8. Windows 中设置 Android NDK

在 Windows PowerShell 中设置：

$env:ANDROID_NDK="C:\Users\<用户名>\AppData\Local\Android\Sdk\ndk\<NDK版本号>"

例如：

$env:ANDROID_NDK="C:\Users\93739\AppData\Local\Android\Sdk\ndk\29.0.14206865"

检查路径是否正确：

Test-Path "$env:ANDROID_NDK\build\cmake\android.toolchain.cmake"

如果输出：

True

说明路径正确。

为什么要设置这个变量：

CMake 需要通过 Android NDK 提供的 toolchain 文件来生成 Android 平台的交叉编译配置。否则 CMake 不知道应该使用哪个 clang、哪个 sysroot，以及目标 ABI 是什么。

是否换新模型需要重新执行：

不需要。这个步骤只和 Android 编译环境有关，和模型无关。

# 9. Windows 中编译 Android 版 llama.cpp

在 Windows 的 llama.cpp 目录中执行：

cd $env:USERPROFILE\llama.cpp

Remove-Item -Recurse -Force build-android -ErrorAction SilentlyContinue

cmake -S . -B build-android -G "Ninja" `
  -DCMAKE_BUILD_TYPE=Release `
  -DBUILD_SHARED_LIBS=OFF `
  -DGGML_OPENMP=OFF `
  -DCMAKE_TOOLCHAIN_FILE="$env:ANDROID_NDK\build\cmake\android.toolchain.cmake" `
  -DANDROID_ABI=arm64-v8a `
  -DANDROID_PLATFORM=android-28

cmake --build build-android -j4

为什么要在 Windows 中编译：

当前安装的是 Windows 版 Android NDK，其中预编译工具链位于 windows-x86_64。如果在 WSL 中直接调用这个 NDK，CMake 会寻找 linux-x86_64 工具链，从而报错。因此这里选择在 Windows 中使用 Windows NDK 编译 Android 可执行文件。

为什么设置 ANDROID_ABI=arm64-v8a：

绝大多数现代 Android 手机都是 ARM64 架构，arm64-v8a 是最常用的目标 ABI。

为什么设置 ANDROID_PLATFORM=android-28：

android-28 对应 Android 9，兼容性较好。只要手机 API level 高于或等于 28，通常都可以运行。

为什么设置 BUILD_SHARED_LIBS=OFF：

这样会尽量生成静态链接的可执行文件，减少运行时缺少动态库的问题。之前常见错误包括：

libmtmd.so not found

通过关闭 shared libs 可以减少这类依赖问题。

为什么设置 GGML_OPENMP=OFF：

Android 上 OpenMP 可能需要额外的 libomp.so。如果没有一起打包或设置 LD_LIBRARY_PATH，会报：

libomp.so not found

关闭 OpenMP 后，运行环境更简单，更适合 demo 和初步部署。

是否换新模型需要重新编译：

不需要。模型文件变化不影响可执行程序。只有当你更新了 llama.cpp 源码、修改了算子、加入 Monarch op，或者改变编译选项时，才需要重新编译。

# 10. 将模型从 WSL 拷贝到 Windows


copy "\\wsl$\Ubuntu\home\user\llama.cpp\llama2-7b-q4_k_m.gguf" `
     "$env:USERPROFILE\llama.cpp\llama2-7b-q4_k_m.gguf"

为什么要拷贝到 Windows：

ADB 运行在 Windows 侧，直接从 Windows 路径 push 文件更方便。虽然也可以通过 \\wsl$ 直接访问 WSL 文件，但大文件传输时拷贝到 Windows 本地路径通常更稳定。

是否换新模型需要重新执行：

是。换了模型或量化文件后，需要重新拷贝并 push 到手机。

# 11. 使用 ADB 推送到手机

创建手机端目录：

adb shell "mkdir -p /data/local/tmp/llama && chmod 777 /data/local/tmp/llama"

推送可执行文件：

adb push "$env:USERPROFILE\llama.cpp\build-android\bin\llama-cli" /data/local/tmp/llama/llama-cli
adb push "$env:USERPROFILE\llama.cpp\build-android\bin\llama-bench" /data/local/tmp/llama/llama-bench

推送模型：

adb push "$env:USERPROFILE\llama.cpp\llama2-7b-q4_k_m.gguf" /data/local/tmp/llama/

赋予执行权限：

adb shell "chmod +x /data/local/tmp/llama/llama-cli /data/local/tmp/llama/llama-bench"

为什么放在 /data/local/tmp/llama：

/data/local/tmp 是 Android 上常用于调试和 adb 运行 native 程序的目录。它不需要 root 权限，通常允许写入和执行，适合部署测试型 demo。

是否换新模型需要重新执行：

如果只换模型，只需要重新 push 模型文件。如果重新编译了二进制，则需要重新 push llama-cli 和 llama-bench。

# 12. 在手机端运行推理

进入 adb shell：

```
adb shell
```


在手机端执行：


```
cd /data/local/tmp/llama

./llama-cli \
  -m llama2-7b-q4_k_m.gguf \
  -p "Hello" \
  -n 64 \
  -t 8
```


参数说明：

```
-m  指定模型路径
-p  指定 prompt
-n  指定生成 token 数
-t  指定线程数
```
